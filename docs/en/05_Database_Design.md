## 2. Purpose

This document defines the PostgreSQL database design required for the SentinelAI Phase 1 Authentication MVP.

The Phase 1 database supports:

* User registration
* User login
* Password-hash storage
* Account activation status
* Authenticated user retrieval
* Account creation and update timestamps

The design intentionally excludes log management, AI analysis, alerts, organizations, roles, API keys, and audit logs.

## 4. Database Technology

SentinelAI Phase 1 uses PostgreSQL.

Reasons:

* Strong relational-data support
* Reliable constraints and transactions
* Mature indexing
* Good Python ecosystem support
* Suitable for future production deployment
* Appropriate for authentication and future log metadata
* Supported by managed AWS database services in later phases

## 5. Database Access Design

The backend will use:

* SQLAlchemy 2.x as the ORM and database-access layer
* Alembic for schema migrations
* PostgreSQL as the database engine

The backend must not create or change production tables automatically during application startup.

Schema changes must be managed through version-controlled migrations.

### 7.3 Column Definitions

| Column          | PostgreSQL Type | Nullable | Default           | Constraint  | Purpose                  |
| --------------- | --------------- | -------: | ----------------- | ----------- | ------------------------ |
| `id`            | `UUID`          |       No | Generated UUID    | Primary key | Stable user identifier   |

## 8. Column Design Details

### 8.1 `id`

Recommended type:

```text
UUID
```

Purpose:

* Uniquely identifies the user
* Avoids sequential public identifiers
* Supports future distributed deployment
* Can be safely used as the JWT `sub` claim

The UUID should be generated by the application or PostgreSQL using a supported UUID-generation function.

The implementation decision must remain consistent across models, tests, and migrations.

### 8.2 `username`

Type:

```text
VARCHAR(50)
```

Rules:

* Required
* Unique
* Minimum length: 3 characters
* Maximum length: 50 characters
* Leading and trailing whitespace must be removed
* Case-handling must be defined consistently
* Must not contain control characters

Initial permitted characters:

```text
Letters
Numbers
Underscore
Hyphen
Spaces may be rejected in Phase 1 for simplicity
```

Recommended validation pattern:

```regex
^[A-Za-z0-9_-]{3,50}$
```

The username is not used as the primary login credential in Phase 1.

### 8.3 `email`

Type:

```text
VARCHAR(254)
```

Rules:

* Required
* Unique
* Used as the login identifier
* Leading and trailing whitespace must be removed
* Stored in normalized lowercase form
* Validated by the backend
* Must not be exposed in application logs unnecessarily

Email uniqueness must be enforced by PostgreSQL, not only by application logic.

### 8.4 `password_hash`

Type:

```text
VARCHAR(255)
```

Rules:

* Required
* Stores only the password hash
* Never stores the original password
* Must not be returned by API responses
* Must not be logged
* Must not be included in JWT claims

The current design uses Argon2 password hashing.

The column length provides room for Argon2 encoded hashes and future hashing upgrades.

### 8.5 `is_active`

Type:

```text
BOOLEAN
```

Default:

```text
TRUE
```

Purpose:

* Allows the system to disable an account without deleting it
* Supports future administrative controls
* Allows authentication to be rejected for inactive accounts

A hard delete is not required during Phase 1.

### 8.6 `created_at`

Type:

```text
TIMESTAMPTZ
```

Default:

```text
Current UTC-aware database timestamp
```

Purpose:

* Records when the account was created
* Supports future auditing and account management

### 8.7 `updated_at`

Type:

```text
TIMESTAMPTZ
```

Default:

```text
Current UTC-aware database timestamp
```

Purpose:

* Records the most recent account update

The application or database layer must update this value whenever the user record is changed.

## 10. Case Sensitivity Decision

Email addresses will be normalized to lowercase before storage.

Example:

```text
Sai.Example@Example.com
```

Stored as:

```text
sai.example@example.com
```

This avoids duplicate accounts that differ only by letter case.

Usernames will also be normalized for uniqueness checks.

Recommended Phase 1 behavior:

* Store the normalized username
* Treat usernames as case-insensitive for uniqueness
* Preserve no separate display-name field yet

This may be revisited when profile features are introduced.

## 11. Index Design

Required indexes:

| Index             | Column     | Reason                      |
| ----------------- | ---------- | --------------------------- |
| Primary-key index | `id`       | Automatic user lookup       |
| Unique index      | `email`    | Login lookup and uniqueness |
| Unique index      | `username` | Username uniqueness         |

An additional index on `is_active` is not required during Phase 1 because the table is expected to remain small and most queries will first identify a single user.

Avoid unnecessary indexes because each index increases write and storage cost.

## 12. Proposed PostgreSQL DDL

The following SQL represents the intended logical schema.

It is design documentation, not the final migration file.

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    email VARCHAR(254) NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT uq_users_username UNIQUE (username),
    CONSTRAINT uq_users_email UNIQUE (email)
);
```

The final migration may include PostgreSQL UUID-generation configuration depending on whether identifiers are generated by the application or database.

## 15. Transaction Design

User registration should use a database transaction.

Expected flow:

```text
Begin transaction
    ↓
Normalize input
    ↓
Check existing user
    ↓
Hash password
    ↓
Insert user
    ↓
Commit transaction
```

If insertion fails:

```text
Rollback transaction
```

The database uniqueness constraint remains the final protection against duplicate email or username creation, including concurrent requests.

The backend must handle unique-constraint violations safely.

## 16. Migration Strategy

Alembic will manage database schema changes.

Planned initial migration:

```text
Create users table
Create primary key
Create unique username constraint
Create unique email constraint
Create timestamp columns
```

Rules:

* Migration files must be committed to Git.
* Migrations must be reviewed before execution.
* Migration history must not be rewritten after shared use without a strong reason.
* Schema changes must not be performed manually in production.
* Docker startup must not silently destroy existing database data.

## 17. Seed Data

Phase 1 does not require production seed data.

Optional development seed data may be introduced later through an explicit script.

Seed data must not contain:

* Real passwords
* Real personal email addresses
* Production credentials
* Customer information

## 18. Data Security

The database design must follow these rules:

* Plaintext passwords are never stored.
* Password hashes are never returned to the frontend.
* JWT secrets are not stored in the users table.
* Database credentials are supplied through environment variables.
* The database is not exposed publicly in production.
* Database logs must not contain passwords or hashes.
* Backups must be protected when introduced.
* Least-privilege database users should be used in production.

## 19. Data Retention and Deletion

Phase 1 does not implement user self-deletion.

Future account-deletion requirements may include:

* Soft deletion
* Hard deletion
* Audit retention
* Log ownership handling
* Personal data deletion

These requirements will be designed in a later phase.
